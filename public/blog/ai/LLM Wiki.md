## LLM Wiki란?

[LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)는 Andrej Karpathy가 제안한 **LLM 기반 개인 지식 베이스 구축 패턴**이다. LLM이 단순히 질문에 답하는 것을 넘어, **지속적으로 위키를 생성하고 유지보수**하게 만드는 아이디어다.

```
원본 소스 → LLM 분석 → 위키 페이지 생성/업데이트 → 질의 응답
```

핵심은 RAG(Retrieval-Augmented Generation)와의 차이에 있다. RAG는 매번 원본 문서에서 청크를 검색해 답변을 생성하지만, LLM Wiki는 **누적되는 지식 아티팩트**를 만든다.

---

## 왜 필요한가?

대부분의 LLM 활용은 파일을 업로드하고 질문하는 RAG 방식이다. 이 접근법은 작동하지만 **축적이 없다**.

| 방식 | 한계 |
|---|---|
| RAG | 매 질문마다 처음부터 지식 재발견 |
| 노트 앱 | 사람이 정리해야 해서 결국 방치 |
| 북마크 | 저장만 하고 다시 안 봄 |
| 수동 위키 | 유지보수 부담이 커서 포기 |

Karpathy의 진단은 명확하다:

> "지식 베이스 유지의 어려운 부분은 읽기나 사고가 아니라 **정리 작업(bookkeeping)**이다."

LLM이 바로 이 정리 작업을 대신한다. 사람은 소스를 큐레이션하고 질문하며, LLM은 요약, 교차 참조, 분류, 유지보수를 담당한다.

---

## 아키텍처

LLM Wiki는 세 개의 레이어로 구성된다.

```
┌─────────────────────────────────────────┐
│            Schema (설정)                 │
│  CLAUDE.md / AGENTS.md                  │
│  위키 구조, 컨벤션, 워크플로우 정의       │
├─────────────────────────────────────────┤
│            Wiki (생성물)                 │
│  마크다운 파일 컬렉션                     │
│  요약, 엔터티, 개념, 비교, 인덱스         │
│  → LLM이 완전히 소유하는 레이어           │
├─────────────────────────────────────────┤
│         Raw Sources (원본)               │
│  논문, 기사, 이미지, 데이터 파일           │
│  → 불변, 사람이 큐레이션                  │
└─────────────────────────────────────────┘
```

### Raw Sources (원본 소스)

사람이 큐레이션하는 불변 문서들이다. 논문, 기사, 이미지, 데이터 파일이 여기에 속한다. **소스 오브 트루스(source of truth)** 역할을 한다.

### Wiki (위키)

LLM이 생성하고 유지하는 마크다운 파일 컬렉션이다. 요약 페이지, 엔터티 페이지, 개념 페이지, 비교 페이지, 개요 페이지 등으로 구성된다. **LLM이 이 레이어를 완전히 소유**한다.

### Schema (스키마)

`CLAUDE.md`나 `AGENTS.md` 같은 설정 문서로, 위키의 구조, 네이밍 컨벤션, 워크플로우를 정의한다. LLM이 일관되게 동작하도록 가이드라인을 제공한다.

## 핵심 오퍼레이션

### 1. Ingest (수집)

새로운 소스가 들어오면 LLM이 처리한다.

```
새 소스 추가
  → LLM이 읽고 분석
  → 사용자와 핵심 내용 논의
  → 요약 페이지 작성
  → 인덱스 업데이트
  → 관련 엔터티/개념 페이지 수정
  → 로그 기록
```

하나의 소스가 보통 **10-15개의 위키 페이지**에 영향을 준다. 단순 저장이 아니라 기존 지식과의 교차 참조, 모순 감지, 통합이 일어난다.

### 2. Query (질의)

사용자가 위키에 대해 질문하면 LLM이 관련 페이지를 검색하고 답변을 합성한다.

```
질문 → 관련 위키 페이지 탐색 → 답변 합성 (인용 포함)
```

좋은 답변은 새로운 위키 페이지로 다시 저장될 수 있다. 마크다운, 테이블, 슬라이드, 차트 등 다양한 형태로 출력 가능하다.

### 3. Lint (점검)

주기적인 건강 검진이다.

| 점검 항목 | 설명 |
|---|---|
| 모순 감지 | 페이지 간 상충하는 내용 식별 |
| 오래된 주장 | 새로운 소스에 의해 대체된 정보 표시 |
| 고아 페이지 | 인바운드 링크가 없는 페이지 발견 |
| 누락 개념 | 전용 페이지가 없는 중요 개념 식별 |
| 데이터 갭 | 정보가 부족한 영역 표시 |

## 인덱싱 시스템

### index.md (콘텐츠 중심)

모든 위키 페이지를 **카테고리별로 정리**한 목차다. 링크, 요약, 메타데이터를 포함한다. 소스를 수집할 때마다 업데이트되며, LLM이 관련 페이지를 빠르게 찾도록 돕는다.

### log.md (시간순)

**언제 무엇이 일어났는지** 기록하는 추가 전용(append-only) 로그다. 일관된 접두사를 사용해 파싱이 쉽다.

```bash
# 최근 5개 항목 확인
grep "^## \[" log.md | tail -5
```

## 활용 사례

| 영역 | 활용 방법 |
|---|---|
| **개인 성장** | 일기, 목표, 건강 기록에서 패턴 추출 |
| **연구** | 수주~수개월간 논문을 읽으며 포괄적 위키 구축 |
| **독서** | 챕터별 정리, 캐릭터/테마 페이지 자동 생성 |
| **비즈니스** | Slack 스레드, 미팅 기록에서 내부 위키 유지 |
| **분석** | 경쟁사 인텔리전스, 실사, 여행 계획, 강의 노트 |

## 실전 운영 팁

커뮤니티에서 수주간 실제 운영하며 축적된 인사이트들이다.

### 소스 분류 후 처리

문서를 처리하기 전에 **타입을 먼저 분류**한다. 논문인지, 블로그인지, 미팅 노트인지에 따라 추출 전략이 달라진다. 토큰 효율이 높아진다.

### 타입별 템플릿

범용 템플릿 대신 **엔터티 타입별 전용 템플릿**을 사용한다. 인물 페이지, 개념 페이지, 도구 페이지 각각의 구조가 다르다. 구조적 일관성이 유지된다.

### 토큰 예산 관리

컨텍스트를 4단계로 나눠 **점진적 공개(progressive disclosure)** 방식을 적용한다.

```
L0: 한 줄 요약
L1: 핵심 포인트 (3-5개)
L2: 상세 분석
L3: 전체 내용 + 인용
```

### 이중 출력 원칙

모든 작업이 **사용자 요청 출력 + 위키 업데이트** 두 가지를 동시에 생산한다. 지식 증발을 방지한다.

### 출처 추적 (Provenance)

어떤 소스가 어떤 주장을 만들었는지, 콘텐츠 해시와 함께 기록한다. 시간이 지나 소스가 업데이트되면 **어떤 위키 페이지가 오래됐는지** 자동 감지할 수 있다.

## 도구 생태계

### Obsidian 연동

| 기능 | 설명 |
|---|---|
| Web Clipper | 웹 기사를 마크다운으로 빠르게 변환 |
| 로컬 이미지 | URL 의존 없이 콘텐츠 보존 |
| 그래프 뷰 | 위키 구조와 연결 관계 시각화 |
| Dataview | YAML 프론트매터 기반 동적 테이블 |
| Marp | 마크다운 기반 슬라이드 제작 |

### 검색 도구

위키가 커지면 전용 검색이 필요하다. [QMD](https://github.com/yagueto/qmd) 같은 도구가 하이브리드 BM25/벡터 검색과 LLM 재랭킹을 제공한다. MCP 서버 연동으로 LLM이 네이티브하게 검색을 활용할 수 있다.

### 버전 관리

위키를 Git 리포지토리로 관리하면 **변경 이력과 협업**이 가능해진다.

## RAG vs LLM Wiki

| 항목 | RAG | LLM Wiki |
|---|---|---|
| 지식 축적 | 없음 (매번 재검색) | 위키에 누적 |
| 교차 참조 | 질의 시점에 발견 | 수집 시점에 생성 |
| 모순 감지 | 안 됨 | Lint로 주기적 점검 |
| 유지보수 비용 | 낮음 | LLM이 담당 |
| 인간 개입 | 질의만 | 소스 큐레이션 + 질의 |
| 적합한 규모 | 단발성 질의 | 장기적 지식 구축 |

RAG가 "검색 엔진"이라면, LLM Wiki는 **"전담 리서치 어시스턴트"**에 가깝다.

## Memex와의 연결

1945년 Vannevar Bush가 제안한 [Memex](https://en.wikipedia.org/wiki/Memex) 개념과 직접적으로 연결된다. 개인이 큐레이션한 지식 저장소에서 문서 간 연상적 연결(associative trails)을 만드는 비전이다.

Bush의 비전과 LLM Wiki의 차이는 단 하나: **유지보수**. 80년 전에는 사람이 해야 했던 정리 작업을 이제 LLM이 대신한다.

---

## 한계와 고려사항

- **검증 책임**: LLM은 인용 없이 합성할 수 있다. 사실 확인은 사람의 몫이다
- **환각 위험**: 위키 페이지 자체가 LLM이 생성한 것이므로, 원본 소스 대조가 필수다
- **온톨로지 설계**: 자유로운 링크보다 명시적 엔터티 타입과 관계 타입을 정의해야 깔끔한 위키가 나온다
- **프라이버시**: 민감한 자료는 로컬 모델(Ollama 등)로 처리하고, 일반 자료만 클라우드 LLM에 보내는 아키텍처가 필요할 수 있다

## 정리

LLM Wiki는 LLM을 "질문-답변 기계"에서 **"지식 관리 엔진"**으로 격상시키는 패턴이다.

| 항목 | 내용 |
|---|---|
| 제안자 | Andrej Karpathy |
| 형태 | 마크다운 기반 위키 + LLM 자동화 |
| 핵심 오퍼레이션 | Ingest, Query, Lint |
| 레이어 | Raw Sources → Wiki → Schema |
| 도구 | Obsidian, QMD, Git, MCP |
| 원본 | [GitHub Gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) |

한 줄 요약: **LLM에게 위키 유지보수를 맡기면, 사람은 읽고 생각하는 데만 집중할 수 있다.**
