## OpenViking이란?

OpenViking은 **AI 에이전트를 위해 설계된 오픈소스 컨텍스트 데이터베이스**입니다.

보통 에이전트를 만들다 보면 메모리는 따로, 리소스 문서는 따로, 스킬은 또 따로 흩어져 관리되기 쉽습니다. OpenViking은 이 문제를 **파일시스템 패러다임**으로 정리합니다. 즉, 메모리·리소스·스킬을 하나의 계층 구조 안에서 다루게 만들어, 에이전트가 문맥을 더 구조적으로 읽고 추적할 수 있게 돕습니다.

프로젝트 소개 문구대로라면 OpenViking은 단순 벡터 저장소가 아니라, **에이전트의 뇌를 구성하는 문맥 계층 전체를 관리하는 기반 시스템**에 가깝습니다.

- GitHub: https://github.com/volcengine/OpenViking
- 성격: 오픈소스 컨텍스트 DB / Agent memory platform
- 주 용도: 메모리, 리소스, 스킬, 세션 문맥 관리

## 왜 주목받는가?

OpenViking이 흥미로운 까닭은, 기존 RAG가 가진 몇 가지 답답함을 정면으로 건드리기 때문입니다.

### 1. 문맥을 "조각"이 아니라 "디렉터리"로 본다

전통적인 RAG는 문서를 잘게 쪼개 벡터로 넣고, 질의가 오면 비슷한 조각 몇 개를 꺼내는 방식이 많습니다. 이 방식은 간단하지만, 정보가 **어느 문맥 안에 속했는지**를 이해하기 어렵습니다.

OpenViking은 이를 바꾸어 **viking:// 형태의 가상 파일시스템** 안에 문맥을 배치합니다.

```text
viking://
├── resources/
├── user/
└── agent/
```

즉,
- `resources/`에는 프로젝트 문서나 외부 자료,
- `user/`에는 사용자 메모리,
- `agent/`에는 스킬·지침·작업 메모리
를 넣어, 에이전트가 문맥을 경로 단위로 탐색하게 합니다.

### 2. L0/L1/L2 계층으로 토큰 낭비를 줄인다

OpenViking은 문맥을 저장할 때 한 번에 전부 프롬프트에 넣는 대신, 단계별로 나눕니다.

- **L0 (Abstract)**: 한 줄 요약
- **L1 (Overview)**: 핵심 개요
- **L2 (Details)**: 원문 전체

이 구조 덕분에 에이전트는 먼저 가벼운 요약을 보고, 필요할 때만 더 깊은 계층으로 내려갑니다. 결국 **컨텍스트 윈도우를 아끼고**, 동시에 더 정확한 탐색이 가능해집니다.

### 3. 검색 경로가 보여서 디버깅하기 쉽다

RAG는 결과가 왜 그렇게 나왔는지 설명하기 어려운 경우가 많습니다. OpenViking은 디렉터리 재귀 탐색 구조를 내세우며, **검색이 어떤 경로를 따라 내려갔는지**를 관찰 가능하게 하려 합니다.

이 점은 특히 실무에서 중요합니다. 에이전트가 헛소리를 했을 때,
- 어떤 문맥을 읽었는지,
- 어떤 단계에서 엇나갔는지,
- 검색 기준을 어떻게 바꿔야 하는지
를 훨씬 잡아내기 쉬워집니다.

## 핵심 개념 정리

### 1. Filesystem Management Paradigm

OpenViking의 가장 중요한 아이디어입니다.

문맥을 추상적인 벡터 조각이 아니라 **파일과 디렉터리처럼 관리**합니다. 개발자 입장에서는 그냥 저장 구조를 정리하는 문제처럼 보이지만, 에이전트 입장에서는 이것이 꽤 큽니다.

왜냐하면 에이전트는
- `ls`처럼 훑어보고,
- `find`처럼 찾고,
- 특정 디렉터리로 더 깊게 들어가며,
- 필요한 원문만 집어 읽는
방식으로 문맥을 다룰 수 있기 때문입니다.

### 2. Directory Recursive Retrieval

검색도 단순 유사도 검색으로 끝내지 않습니다.

큰 흐름은 이렇습니다.
1. 질의 의도를 분석한다.
2. 관련성이 높은 디렉터리를 먼저 찾는다.
3. 그 안을 다시 탐색한다.
4. 필요하면 하위 디렉터리까지 재귀적으로 내려간다.
5. 가장 적합한 문맥을 모은다.

이 방식의 장점은, 단순히 "비슷한 문장"이 아니라 **문서가 놓인 위치와 구조**까지 함께 고려한다는 데 있습니다.

### 3. Self-Iterating Memory

OpenViking은 세션이 끝난 뒤 작업 결과와 사용자 피드백을 바탕으로, 장기 기억을 갱신하는 흐름도 제시합니다.

즉 단순 로그 저장이 아니라,
- 사용자 취향 메모리,
- 에이전트 작업 기억,
- 도구 사용 이력,
- 반복되는 패턴
을 장기적으로 축적하는 쪽을 지향합니다.

## 에이전트 하네스 아키텍처 관점에서 보면

전하께서 특히 보라 하신 대목이 바로 여기이옵니다.

OpenViking 자체는 챗 UI가 중심인 제품이라기보다, **에이전트 하네스 아래에 깔리는 문맥 계층**으로 보는 것이 더 정확합니다.

### 구조를 단순화하면

```text
사용자 / 메신저 / 앱
        ↓
에이전트 하네스 (planner / tool runner / session manager)
        ↓
OpenViking (context DB / memory / resource / skill store)
        ↓
모델 호출 / 검색 / 실행
```

즉 역할을 나누면 다음과 같습니다.

- **에이전트 하네스**
  - 사용자의 요청을 해석
  - 작업을 계획
  - 어떤 도구를 쓸지 결정
  - 어떤 문맥을 불러올지 판단

- **OpenViking**
  - 하네스가 참조할 메모리와 리소스를 구조적으로 저장
  - 문맥을 L0/L1/L2 계층으로 제공
  - 검색 경로와 결과를 더 잘 추적할 수 있게 함

### 왜 하네스와 잘 맞는가?

에이전트 하네스가 강해질수록 필요한 것은 더 많은 프롬프트가 아니라, **더 좋은 문맥 운영 시스템**입니다.

OpenViking은 바로 이 부분을 겨냥합니다.

- 장기 메모리 관리
- 프로젝트 문서 저장소화
- 사용자/에이전트 구분 메모리
- 스킬 및 지침 저장
- 재귀 검색 기반 컨텍스트 로딩

즉 OpenViking은 스스로 모든 걸 하는 만능 에이전트라기보다,
**OpenClaw 같은 상위 에이전트 하네스와 결합했을 때 힘이 커지는 하부 인프라**라고 보는 편이 맞습니다.

### OpenClaw와의 연결성

프로젝트 소개에는 OpenClaw 예시가 직접 언급됩니다. 또 OpenClaw 메모리 플러그인 예시와 성능 비교도 공개하고 있습니다.

핵심 메시지는 명확합니다.
- 기본 메모리보다 더 구조적인 문맥 계층을 제공하고,
- 토큰 비용을 줄이면서,
- 작업 성공률을 높이겠다는 것입니다.

즉 OpenViking은 **에이전트 하네스의 검색·메모리 층을 교체하거나 보강하는 아키텍처 부품**으로 이해하면 가장 쉽습니다.

## 설치와 실행

### 요구사항

- Python 3.10+
- Go 1.22+
- C++ 컴파일러
- macOS / Linux / Windows 지원

### 설치 예시

```bash
pip install openviking --upgrade --force-reinstall
```

또는 CLI 설치:

```bash
curl -fsSL https://raw.githubusercontent.com/volcengine/OpenViking/main/crates/ov_cli/install.sh | bash
```

### 서버 실행 예시

```bash
openviking-server
```

또는 백그라운드 실행 후:

```bash
ov status
ov add-resource https://github.com/volcengine/OpenViking
ov ls viking://resources/
ov tree viking://resources/volcengine -L 2
ov find "what is openviking"
```

즉 설치 후에는 단순 라이브러리 호출만이 아니라, **컨텍스트 서버처럼 띄워두고 에이전트가 참조하게 하는 방식**도 가능합니다.

## 어떤 사람에게 맞는가?

### 잘 맞는 경우

- 에이전트의 장기 메모리가 자꾸 깨지는 경우
- RAG 결과가 들쭉날쭉해서 디버깅이 어려운 경우
- 프로젝트 문서/사용자 메모리/스킬을 한데 묶고 싶은 경우
- OpenClaw, OpenCode, 자체 에이전트 하네스 등에 더 강한 문맥 계층을 붙이고 싶은 경우

### 덜 맞는 경우

- 그냥 간단한 문서 검색 챗봇 하나면 충분한 경우
- 벡터DB + 임베딩만으로도 요구사항이 충분히 해결되는 경우
- 운영 복잡도를 최소화해야 하는 아주 작은 프로젝트

즉 OpenViking은 **"에이전트를 제대로 운영하려는 팀"**에게 더 맞고, 아주 가벼운 RAG 실험에는 다소 무거울 수 있사옵니다.

## 한 줄 총평

OpenViking은 "AI 에이전트용 RAG" 정도로 축소해서 보면 아쉬운 프로젝트입니다.

오히려 이것은
**에이전트가 기억·리소스·스킬을 파일시스템처럼 다루게 만드는 컨텍스트 운영체제에 가까운 시도**입니다.

에이전트 하네스 아키텍처를 진지하게 다루는 사람이라면,
OpenViking은 한 번쯤 구조를 뜯어볼 가치가 충분한 프로젝트입니다.
